# Plan: add VBOOT support to `util/cbfstool-rs`

## Goals

Add enough VBOOT support to `cbfstool-rs build-image` that the manifest path can
assemble a bootable vboot image without falling back to a long sequence of
mutable `cbfstool` operations. VBOOT is part of the manifest image-assembly plan,
not a separate replacement path, because VBOOT slotting is the build flow that
requires explicit CBFS FMAP regions (`COREBOOT`, `FW_MAIN_A`, and `FW_MAIN_B`).
The implementation should keep the generated manifest as the source of truth for
image contents and should update verification metadata only after final image
contents are known.

This plan intentionally separates three related but different pieces of
"vboot support":

1. **CBFS verification integration** (`CONFIG_CBFS_VERIFICATION` and
   `CONFIG_VBOOT_CBFS_INTEGRATION`): per-file hash attributes, CBFS metadata
   hashes, FMAP hashes, and bootblock metadata-anchor patching.
2. **VBOOT firmware image data**: GBB, RO/RW FWID regions, shared-data region,
   slot-aware CBFS placement in `COREBOOT`, `FW_MAIN_A`, and `FW_MAIN_B`.
3. **VBOOT signing/GSCVD artifacts**: VBLOCK_A/B firmware preambles, GBB
   generation/setup, and RO_GSCVD generation. These are currently produced by
   `futility` and should initially remain delegated to `futility` from a
   manifest-driven post-assembly step.

## Current build behavior to match

### Slot-aware file placement

`src/security/vboot/Makefile.mk` defines `VBOOT_PARTITIONS` and
`regions-for-file`. With vboot enabled, most CBFS files are installed into the
RO CBFS plus every enabled RW slot:

- always: `COREBOOT`
- optional RW A: `FW_MAIN_A`
- optional RW B: `FW_MAIN_B`

Special names are RO-only (`%/verstage`, `intel_fit`, `fspt.bin`, `locales`,
etc.), and Kconfig lists can force RO-only, RW-only, RWA-only, or RWB-only
placement.

The manifest generator should emit explicit `regions` for each entry; the Rust
builder should not reimplement Kconfig policy.

### VBOOT raw regions

The Makefile currently writes these after `coreboot.rom` exists:

- `GBB`: generated by `futility gbb_utility`, then written with
  `cbfstool write -u -r GBB -i 0`.
- `RO_FRID`: model + firmware ID version NUL-terminated string.
- `RW_FWID_A`, `RW_FWID_B`: same FWID string when slots are enabled.
- `SHARED_DATA`: one NUL byte when the FMAP contains the region.

These map cleanly to manifest raw-region writes, but GBB generation itself is a
new manifest operation unless the build continues to pre-generate `gbb.region`.

### CBFS verification metadata

C `cbfstool` does the following when a metadata-hash anchor is present:

- Finds `metadata_hash_anchor` by scanning the `BOOTBLOCK` FMAP region, or the
  `bootblock`/`apu/amdfw` CBFS entry when there is no separate `BOOTBLOCK`
  region.
- Uses the hash algorithm already stored in the anchor.
- Excludes these CBFS types from per-file hashes: `bootblock`, `cbfs header`,
  `intel_fit`, `amdfw`.
- Adds a `CBFS_FILE_ATTR_TAG_HASH` attribute to each verified file containing
  serialized `struct vb2_hash`: three reserved zero bytes, one algorithm byte,
  followed by the digest.
- Computes the CBFS metadata hash by walking CBFS entries and hashing each
  non-empty entry's serialized metadata from the CBFS file header through
  `data_offset`, including filename, attributes, and padding before data.
- Computes the FMAP hash over the serialized FMAP structure and stores it after
  the CBFS metadata hash in the anchor.
- For special bootblock container formats, applies `platform_fixups.c` hash
  fixups after patching the anchor.

The manifest builder can do this more cleanly by assembling the full image in
memory, then doing one final verification pass.

### Firmware signing

`futility sign` recognizes a BIOS image by FMAP regions `GBB`, `FW_MAIN_A`, and
`VBLOCK_A`. It signs `FW_MAIN_A` into `VBLOCK_A` and, when present, `FW_MAIN_B`
into `VBLOCK_B`. With `CONFIG_VBOOT_CBFS_INTEGRATION`, it requires a valid CBFS
metadata hash in the RW slot instead of signing the full slot body. Without CBFS
integration it truncates RW CBFS before signing.

Initial `cbfstool-rs` support should call `futility sign` after native CBFS
verification metadata has been finalized. Native vboot key parsing/signing is a
larger follow-up project.

## Manifest schema extensions

Add a top-level `vboot` object. Keep concrete generated paths explicit so the
manifest remains deterministic and reviewable.

```json
{
  "version": 1,
  "vboot": {
    "partitions": {
      "ro": "COREBOOT",
      "rw": ["FW_MAIN_A", "FW_MAIN_B"]
    },
    "fwid": {
      "model": "MODEL",
      "version": "1234",
      "regions": ["RO_FRID", "RW_FWID_A", "RW_FWID_B"]
    },
    "gbb": {
      "region": "GBB",
      "source": "build/gbb.region"
    },
    "shared_data": {
      "region": "SHARED_DATA",
      "enabled": true
    },
    "cbfs_verification": {
      "enabled": true,
      "cbfs_regions": ["COREBOOT", "FW_MAIN_A", "FW_MAIN_B"],
      "anchor_containers": [
        { "kind": "bootblock-region", "region": "BOOTBLOCK" },
        { "kind": "cbfs-entry", "region": "COREBOOT", "name": "bootblock" }
      ],
      "platform_fixups": "auto"
    },
    "signing": {
      "enabled": true,
      "tool": "futility",
      "signprivate": "keys/firmware_data_key.vbprivk",
      "keyblock": "keys/firmware.keyblock",
      "kernelkey": "keys/kernel_subkey.vbpubk",
      "version": 1,
      "flags": 0
    },
    "gscvd": {
      "enabled": false,
      "tool": "futility",
      "board_id": "...",
      "root_pubkey": "...",
      "platform_privkey": "...",
      "platform_keyblock": "...",
      "ranges": [
        { "region": "GBB" },
        { "region": "BOOTBLOCK", "size_from": "build/cbfs/fallback/bootblock.bin" },
        { "cbfs_region": "COREBOOT", "entry": "bootblock" }
      ]
    }
  }
}
```

Notes:

- `cbfs[].regions` remains the canonical way to describe slot placement.
  `vboot.partitions` is for validation, summaries, and signing defaults.
- `vboot.gbb.source` supports the bridge path where Make/futility still creates
  the GBB blob. A later `vboot.gbb.generate` object can model HWID, rootkey,
  recoverykey, flags, BMPFV, and sizes.
- `vboot.signing.tool = "futility"` is explicit. This prevents pretending that
  Rust natively understands all vboot key and preamble formats.
- Keep `raw_regions` support for generic writes; `vboot.fwid`, `gbb`, and
  `shared_data` are convenience/validation layers that expand to raw writes.

## Implementation phases

### Phase 1: native hash primitives and serialized vboot hash types

Add a small module, likely `src/vboot/hash.rs` or `src/cbfs/verification.rs`:

- `enum HashAlgorithm { Sha1 = 1, Sha256 = 2, Sha512 = 3, Sha224 = 4, Sha384 = 5 }`
- digest sizes and name parsing compatible with vboot names and numeric IDs.
- `digest(algorithm, data) -> Vec<u8>` using RustCrypto crates:
  - `sha1`
  - `sha2` for SHA-224/256/384/512
- `Vb2Hash { algorithm, digest }` serialization:
  - `[0, 0, 0, algorithm as u8] + digest`
  - length is `4 + digest_size`, matching `offsetof(struct vb2_hash, raw) + size`.

Update `AttributeBuilder::hash()` so it serializes the actual vboot layout, not
a big-endian `u32` algorithm. Add tests for SHA-1/SHA-256/SHA-512 serialized
lengths and bytes.

### Phase 2: add explicit hash attributes when writing CBFS entries

Extend `AddEntryOptions` and manifest CBFS entries with an optional hash policy:

```rust
hash: Option<HashAlgorithm>
```

Behavior:

- If `hash` is set, compute the digest of the final stored file data after ELF
  conversion and compression decisions, then append a hash attribute.
- Reject hash attributes on excluded types (`bootblock`, `cbfs header`,
  `intel_fit`, `amdfw`) unless an explicit force/debug option is added.
- Ensure placement calculations include the hash attribute size before locating
  free space.

For normal `cbfstool-rs add`, add CLI parity for C cbfstool's
`-A/--hash-algorithm` option. For manifest builds, this will mostly be driven by
`vboot.cbfs_verification` rather than per-entry JSON.

### Phase 3: final image verification pass in `build-image`

Add a finalization step after all CBFS entries, integer entries, raw writes, and
FIT operations are complete, but before writing the output file:

1. Discover anchors:
   - separate `BOOTBLOCK` and `BOOTBLOCK_B` regions;
   - otherwise `bootblock` or `apu/amdfw` entry in `COREBOOT` for primary;
   - `COREBOOT_B`/`BOOTBLOCK_B` for top-swap secondary.
2. Read algorithm from each anchor; if the manifest provides an algorithm,
   validate it matches the embedded anchor.
3. For every verified CBFS region, rebuild files that need hash attributes or
   patch reserved hash padding in-place.
4. Compute CBFS metadata hash with C-compatible semantics:
   - walk headers at CBFS alignment;
   - ignore deleted/null entries;
   - hash exactly `entry.metadata_len()` bytes starting at `entry.offset()`.
5. Compute FMAP hash over `fmap` bytes, not over arbitrary region contents.
6. Patch anchors once with CBFS metadata hash and FMAP hash.
7. Apply platform fixups where needed.

Important design point: adding hash attributes changes CBFS metadata length and
possibly placement. The easiest reliable implementation is to make hash
attributes part of entry construction before placement. The finalization pass
should not grow entries unless the manifest explicitly reserved padding for it.
Therefore Phase 3 should make `build-image` decide the verification algorithm
before adding entries to verified regions and pass that algorithm into each entry
write.

### Phase 4: verification and print support

Add `cbfstool-rs print -v` parity for verification state:

- parse hash attributes;
- verify per-file hashes;
- recompute metadata hash;
- compare against the anchor for primary/secondary regions;
- print `[METADATA HASH]` and `[CBFS VERIFICATION (...)]` lines compatible with
  C cbfstool where practical.

Also add a dedicated command for CI and manifest validation:

```sh
cbfstool-rs verify --region COREBOOT [--region FW_MAIN_A] image.rom
```

This should return non-zero on missing file hashes, invalid file hashes, missing
anchors when required, invalid metadata hash, or invalid FMAP hash.

### Phase 5: VBOOT raw data helpers

Implement manifest expansion for common vboot raw regions:

- `vboot.fwid`: generate `model + version + NUL` and write it to all listed
  regions with fill-upward semantics.
- `vboot.shared_data`: write one NUL byte to `SHARED_DATA` if enabled or if
  region exists and policy is `auto`.
- `vboot.gbb.source`: write an existing `gbb.region` to `GBB`.

Then optionally add GBB generation:

- initial bridge: `vboot.gbb.generate.tool = "futility"` invokes
  `futility gbb_utility` to create/setup the blob in a temp file;
- later native parser/writer for GBB header, HWID, flags, root key, recovery
  key, and BMPFV.

### Phase 6: signing and GSCVD as manifest-managed post-processes

Add a controlled subprocess layer, separate from generic shell commands:

```rust
trait PostProcessor {
    fn run(&self, image_path: &Path, manifest: &Manifest) -> Result<()>;
}
```

Initial post-processors:

- `FutilitySign`: runs the equivalent of current Makefile
  `futility sign --signprivate ... --keyblock ... --kernelkey ... --version ... --flags ... image`.
- `FutilityGscvd`: computes manifest ranges, then runs `futility gscvd` with
  `-G`, `-R`, board ID, root key, platform key, and keyblock.

To avoid mutable intermediate output becoming the main artifact, write the
assembled image to a same-directory temporary file, run post-processors on that
file, run native verification afterward, then atomically rename to final output.

### Phase 7: manifest generation bridge

Update the Make bridge that emits `coreboot.manifest.json`:

- emit vboot-expanded `regions` for every CBFS entry using existing
  `regions-for-file`;
- emit `vboot.fwid`, `gbb`, `shared_data`, `signing`, and `gscvd` sections from
  existing Kconfig variables;
- keep generated raw files (`gbb.region`, `fwid.region`) initially to reduce
  risk, then remove them once native generation is available.

## Testing strategy

### Unit tests

- vboot hash algorithm parsing and digest sizes.
- `struct vb2_hash` serialization for SHA-1/SHA-256/SHA-512.
- CBFS hash attribute parsing and validation.
- C-compatible CBFS metadata hash walk, including:
  - normal file;
  - deleted/null entries ignored;
  - compression/stage/position/alignment attributes included in metadata hash;
  - excluded types do not need file hashes.
- anchor scanning and patching for:
  - separate `BOOTBLOCK` region;
  - `bootblock` CBFS entry;
  - `apu/amdfw` CBFS entry.

### Golden tests against C cbfstool/futility

Build tiny synthetic images and compare semantics:

1. C cbfstool adds files with `-A sha256`; Rust prints and verifies them.
2. Rust builds files with SHA-256 hash attributes; C cbfstool prints them as
   valid.
3. Rust computes metadata hash; C cbfstool `print -v` reports valid.
4. C cbfstool mutates and updates anchor; Rust verifier agrees.
5. Manifest-built `FW_MAIN_A` and `FW_MAIN_B` pass `futility sign` and
   `futility show`.

### Board-level comparison builds

Use representative configurations:

- RO-only vboot image.
- RW_A image.
- RW_AB image.
- `VBOOT_CBFS_INTEGRATION=y` image.
- top-swap / secondary bootblock image if available.
- GSCVD-enabled Ti50 image.

Compare FMAP layout, CBFS listings, verification state, VBLOCK presence, and
futility validation. Byte identity is a bonus; semantic equivalence is the goal.

## Risks and open questions

- **Growing entries during finalization:** hash attributes must be accounted for
  before placement or must use reserved padding. Prefer deciding hash policy
  before writing entries.
- **Existing `AttributeBuilder::hash()` layout:** it currently accepts a `u32`
  algorithm and writes big-endian bytes; vboot expects three reserved bytes plus
  one algorithm byte. Fix this before relying on native hash attributes.
- **Platform fixups:** C cbfstool updates Qualcomm/MediaTek embedded bootblock
  hashes after anchor patching. Rust needs parity or an explicit unsupported
  error for images requiring those fixups.
- **Native signing scope:** firmware signing depends on vboot keyblock, packed
  key, preamble, and RSA formats. Use `futility` first; consider native support
  only after manifest assembly is stable.
- **Top-swap semantics:** primary and secondary anchors use different containers
  and must not share cached metadata state.
- **`CONFIG_VBOOT_CBFS_INTEGRATION`:** signing must happen after Rust metadata
  hashes are valid; otherwise `futility sign` will reject the image.
- **Hash crate licensing/supply:** use normal RustCrypto crates under compatible
  licenses and keep them optional if the library is used without vboot support.

## Suggested first patch series

1. Add `src/vboot/hash.rs`, SHA dependencies, and tests for digest/serialization. (Started.)
2. Fix `AttributeBuilder::hash()` and add hash attribute parsing helpers. (Serialization fixed; richer parsing remains.)
3. Add `--hash-algorithm` to `add`, `add-payload`, `add-stage`,
   `add-flat-binary`, and manifest CBFS entries. (Started.)
4. Add manifest expansion for FWID/GBB/shared-data writes. (Started.)
5. Add metadata-anchor scanner and C-compatible CBFS metadata hash computation. (Started.)
6. Wire manifest bootblock finalization so an anchor-containing bootblock source can be patched before CBFS insertion, then inserted after the other files are present. (Started for raw bootblock entries.)
7. Complete anchor discovery/patching for existing in-image containers and top-swap secondary containers. (Started: default `COREBOOT`/`BOOTBLOCK`, `COREBOOT_B`/`BOOTBLOCK_B`, CBFS `bootblock`, CBFS `apu/amdfw`, and explicit `anchor_containers` are supported.)
8. Add native `cbfstool-rs verify` for per-file hash attributes only.
9. Wire `vboot.cbfs_verification` into `build-image` so verified regions get
   hash attributes during entry creation and anchors are patched once at the end.
   (Hash attributes are wired when `hash_algorithm` is explicit; deferred bootblock source patching and final in-image anchor patching have started.)
10. Add `futility sign`/`gscvd` manifest post-processors behind explicit `tool` fields.
