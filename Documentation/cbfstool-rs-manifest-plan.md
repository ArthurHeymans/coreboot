# Manifest-driven coreboot image assembly with `cbfstool-rs`

## Problem statement

The current build already has the information needed to assemble CBFS and raw
FMAP contents, but it is encoded as Make variables plus generated shell command
streams. The build creates and mutates `coreboot.pre` / `coreboot.rom` through
many imperative `cbfstool`, `ifittool`, `ifwitool`, and `dd` operations. This is
awkward, hard to validate as data, and especially fragile for CBFS verification:
the verification anchor is patched repeatedly while the image is in intermediate
states.

The goal is to generate a machine-readable manifest describing everything that
must enter the image, then have `util/cbfstool-rs` assemble the final firmware
image from that manifest in one pass.

## Current build flow

The central flow is in `Makefile.mk`:

- `cbfs-files-y += <name>` collects CBFS entries.
- Per-entry variables describe source, type, compression, placement, and extra
  options:
  - `<name>-file`
  - `<name>-type`
  - `<name>-compression`
  - `<name>-position`
  - `<name>-align`
  - `<name>-options`
- `cbfs-files-handler` normalizes those variables into packed records of the
  form:

  ```text
  path|cbfs-name|type|compression|position|align|options
  ```

- `prebuild-files` expands the packed records into many `$(CBFSTOOL) ... add`
  commands.
- `$(obj)/coreboot.pre` creates an FMAP-partitioned image, places the bootblock,
  then applies all CBFS additions.
- `$(obj)/coreboot.rom` creates a full `0xff` ROM, copies `coreboot.pre`, runs
  `INTERMEDIATE` mutation targets, then prints and verifies the result.

Additional Makefile hooks mutate the image outside that central stream:

- `add_intermediate`: FIT updates, EC writes, Intel/AMD firmware writes, CBnT,
  SeaBIOS integer files, etc.
- `build_complete`: vboot GBB/FWID/shared-data writes and other post-build
  region updates.
- `files_added`: signing and final vboot/GSCVD operations.

## Usage categories to model

### Normal CBFS entries

Most entries fit a simple declarative model: source path, CBFS name, type,
compression, regions, placement, and command-specific options. Examples include
stages, payloads, config/revision/build info, FSP, SPD/MRC/VBT, microcode,
Wi-Fi blobs, EC RW hashes, and board data.

### Bootblock placement

Bootblock placement is currently hidden behind architecture/SoC overrides:

- non-x86 raw `write -r BOOTBLOCK`
- x86 `bootblock` CBFS entry with top-aligned placement
- Intel top-swap writes/adds to primary and secondary regions
- AMD reset-vector-in-RAM as either raw AMDFW region writes or `apu/amdfw` CBFS
  entries of type `amdfw`

This should become a first-class manifest section.

### Raw FMAP writes and VBOOT slot data

These are not CBFS entries and should be modeled separately. VBOOT belongs in
this manifest model rather than a separate post-build command stream because it
is the main in-tree user of multiple CBFS FMAP regions (`COREBOOT`,
`FW_MAIN_A`, and `FW_MAIN_B`):

- vboot `GBB`, `RO_FRID`, `RW_FWID_A`, `RW_FWID_B`, `SHARED_DATA`
- vboot slot-aware CBFS placement into explicit per-entry `regions`
- EC firmware regions
- Intel APL `LBP2` and `IFWI`
- AMD signed PSP firmware regions
- HSPHY firmware

### FIT / IFIT updates

FIT mutations currently happen after CBFS population, through `ifittool`:

- FIT pointer updates
- microcode entries
- TXT ACM / IBB entries
- CBnT BPM/KM entries
- PBP entries

These should be declarative manifest operations that run after their referenced
CBFS entries exist.

### Verification anchor updates

The C `cbfstool` scans the bootblock or bootblock-like CBFS entries for the
`metadata_hash_anchor`, adds per-file hashes, computes metadata hashes, computes
FMAP hashes, and patches anchors after individual mutations.

A manifest builder can do this better:

1. assemble the final image in memory;
2. add all entries with their final metadata;
3. compute per-file hashes once;
4. compute CBFS metadata hashes once per verified CBFS region;
5. compute the FMAP hash once;
6. patch all anchors once;
7. write the final ROM.

This avoids requiring a sequence of intermediate CBFS images to be temporarily
verification-valid.

## Proposed manifest format

The manifest should be generated by the build, not usually hand-written. JSON is
a good first target because it is easy to emit, validate, diff, and consume from
Rust.

Example sketch:

```json
{
  "version": 1,
  "output": "build/coreboot.rom",
  "rom_size": 16777216,
  "fmap": {
    "binary": "build/fmap.fmap",
    "cbfs_regions": ["COREBOOT"]
  },
  "cbfs": [
    {
      "name": "fallback/ramstage",
      "source": "build/cbfs/fallback/ramstage.elf",
      "type": "stage",
      "compression": "LZMA",
      "regions": ["COREBOOT"],
      "placement": { "align": 64 },
      "options": {
        "ignore_sections": [".car.data"],
        "xip": false,
        "ibb": false
      }
    }
  ],
  "raw_regions": [
    {
      "region": "GBB",
      "source": "build/gbb.region",
      "mode": "fill-upward",
      "fill": 0
    }
  ],
  "ints": [
    {
      "name": "etc/threads",
      "value": 2,
      "regions": ["COREBOOT"]
    }
  ],
  "bootblock": {
    "source": "build/cbfs/fallback/bootblock.bin",
    "mode": "cbfs-entry",
    "name": "bootblock",
    "type": "bootblock",
    "regions": ["COREBOOT"],
    "placement": { "base": "-file-size" }
  },
  "fit": [
    {
      "op": "set-pointer",
      "fit_region": "COREBOOT",
      "name": "intel_fit",
      "clear": true
    }
  ],
  "verification": {
    "cbfs": true,
    "hash_algorithm": "sha256",
    "patch_anchor": true
  },
  "vboot": {
    "partitions": {
      "ro": "COREBOOT",
      "rw": ["FW_MAIN_A", "FW_MAIN_B"]
    },
    "fwid": {
      "model": "MODEL",
      "version": "1234",
      "regions": ["RO_FRID", "RW_FWID_A", "RW_FWID_B"]
    },
    "gbb": {
      "region": "GBB",
      "source": "build/gbb.region"
    },
    "shared_data": {
      "enabled": true,
      "region": "SHARED_DATA"
    },
    "cbfs_verification": {
      "enabled": true,
      "cbfs_regions": ["COREBOOT", "COREBOOT_B", "FW_MAIN_A", "FW_MAIN_B"],
      "hash_algorithm": "sha256",
      "anchor_containers": [
        { "kind": "bootblock-region", "cbfs_region": "COREBOOT", "region": "BOOTBLOCK" },
        { "kind": "bootblock-region", "cbfs_region": "COREBOOT_B", "region": "BOOTBLOCK_B" },
        { "kind": "cbfs-entry", "cbfs_region": "COREBOOT", "region": "COREBOOT", "name": "bootblock" }
      ]
    }
  }
}
```

## Implementation phases

### Phase 1: Add a manifest consumer to `cbfstool-rs`

Add a command such as:

```sh
cbfstool-rs build-image --manifest build/coreboot.manifest.json
```

Initial responsibilities:

- create a partitioned image from the binary FMAP;
- initialize requested CBFS regions, including vboot RW slot CBFS regions;
- add CBFS entries to each explicit region listed in the manifest;
- add optional vboot per-file hash attributes before placement;
- add integer entries;
- write raw FMAP regions;
- expand vboot convenience records (`GBB`, FWID, shared data) into raw writes;
- place the bootblock;
- emit the final ROM;
- optionally print layout/CBFS summaries.

### Phase 2: Generate the manifest from existing Make data

Do not rewrite every board/SoC Makefile first. Reuse the current `cbfs-files`
collection and replace only the central shell expansion.

A practical bridge is:

1. Make emits a TSV/intermediate list from the packed `cbfs-files` records.
2. `util/cbfstool/gen_manifest.py` or a small Rust helper converts it to JSON.
3. `cbfstool-rs build-image` consumes the JSON.

### Phase 3: Replace central image creation

Change central image assembly from command streams to a single manifest build:

```make
$(obj)/coreboot.rom: $(obj)/coreboot.manifest.json $(CBFSTOOL_RS)
	$(CBFSTOOL_RS) build-image --manifest $<
```

Keep `coreboot.pre` temporarily only for compatibility with existing dependent
rules, then phase it out.

### Phase 4: Convert imperative mutations and VBOOT slot data

Convert these into manifest records:

- SeaBIOS integer entries;
- vboot slot-aware `regions` generation for `COREBOOT`, `FW_MAIN_A`, and `FW_MAIN_B`;
- vboot GBB/FWID/SHARED_DATA writes;
- EC raw region writes;
- AMD signed PSP raw writes and hash entries;
- APL LBP2/IFWI raw writes;
- HSPHY raw writes;
- CBnT BPM/KM updates;
- FIT pointer and FIT entry updates.

### Phase 5: Move verification anchor ownership into `cbfstool-rs`

`cbfstool-rs` should detect anchors, generate hash attributes, compute metadata
hashes with coreboot-compatible semantics, compute FMAP hashes, and patch
primary/secondary anchors once after image assembly.

### Phase 6: Test and compare semantics

Build representative configurations both ways and compare:

- FMAP layout;
- CBFS entry lists, types, compression, placement, and attributes;
- FIT contents;
- CBFS verification state;
- byte-identical output where practical, semantic equivalence otherwise.

Representative cases:

- simple x86 QEMU;
- non-x86 raw bootblock;
- x86 bootblock-in-CBFS;
- CBFS verification enabled;
- vboot GBB/FWID;
- Intel FIT/microcode;
- AMD AMDFW placement;
- top-swap or multiple CBFS regions.

## Known `cbfstool-rs` parity items

Before making the manifest path the default, `cbfstool-rs` needs the workflows
used by the build system rather than every legacy `cbfstool` feature. Important
items are:

- multi-region operations matching `-r A,B` semantics;
- generated position/alignment/IBB/padding/hash attributes;
- CBFS verification anchor detection and patching;
- top-swap master-header pointer updates;
- FIT add-entry and FIT pointer operations;
- `add-stage` parity for ignored sections, XIP placement, and related metadata;
- raw region writes and bootblock placement expressed as manifest operations.
