## Review
- Correct: `src/drivers/amd/atombios/dcn10_calc.c:93-106` packs the basic DCN1 RQ/DLG fields into the bit positions Linux uses: `PREFETCH_SETTINGS` has `VRATIO_PREFETCH` at bits 0..20 and `DST_Y_PREFETCH` at bits 24..31, and `VBLANK_PARAMETERS_0` has VM/row slots at bits 0 and 8 (`/tmp/linux-6.12/drivers/gpu/drm/amd/include/asic_reg/dcn/dcn_1_0_sh_mask.h:9626-9638`). For one linear ARGB8888 plane, the 8 KiB chunk / 1 KiB min-chunk / 2 KiB meta+PTE group constants line up with Linux's RQ setup (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/dml/dml1_display_rq_dlg_calc.c:636-646`, `/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/hubp/dcn10/dcn10_hubp.c:565-573`).
- Correct: Watermark units are mostly right: coreboot converts ns to refclk cycles in `src/drivers/amd/atombios/dcn10_calc.c:18-22`, matching Linux's `wm_ns * refclk_mhz / 1000` clamp for DCN1 watermarks (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/hubbub/dcn10/dcn10_hubbub.c:218-230`).
- Correct: Coreboot programs SAT level and min outstanding in the same form Linux uses: `60 * refclk_mhz` and `68` (`src/drivers/amd/atombios/dcn10.c:376-377`; Linux `/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/hubbub/dcn10/dcn10_hubbub.c:604-607`).

- Blocker: High - `src/drivers/amd/atombios/dcn10.c:362-380` programs all four watermark sets but never toggles `DCHUBBUB_ARB_WATERMARK_CHANGE_REQUEST`. Linux enables the DCN1 workaround `DEGVIDCN10_254` for this resource (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/resource/dcn10/dcn10_resource.c:861-868`) and implements it by writing request 0 then 1 (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/hubbub/dcn10/dcn10_hubbub.c:234-240`). The request bit is bit 8 (`/tmp/linux-6.12/drivers/gpu/drm/amd/include/asic_reg/dcn/dcn_1_0_sh_mask.h:8844-8852`). Minimal fix: add `DCHUBBUB_ARB_WATERMARK_CHANGE_CNTL` at offset `0x051d` and, after watermark/SAT/DF writes, `dcn10_update(..., 1U << 8, 0); dcn10_update(..., 1U << 8, 1U << 8);`.

- Note: Medium - `src/drivers/amd/atombios/dcn10_calc.c:86-113` uses `requests_per_line = ceil(pitch_pixels * 4 / 256)` for `REFCYC_PER_REQ_DELIVERY`, but Linux DML uses `req_per_swath_ub`, derived from viewport width, not pitch: for linear ARGB8888 `blk256_width = 256 / bytes_per_element`, then `swath_width_ub = round_up(vp_width - 1, blk256_width) + blk256_width`, and `req_per_swath_ub = swath_width_ub >> log2_blk256_width` (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/dml/dml1_display_rq_dlg_calc.c:651-660`, `693-696`, `1734-1760`, `1771-1777`). For the checked 1366x768 test mode, coreboot divides by 22 while Linux DML divides by 23, so `DCN_SURF0_TTU_CNTL0` should be `0x08009cb2` instead of `0x0800a3d1`. Minimal fix: compute a separate `requests_per_swath_ub = (round_up(h_active - 1, 64) + 64) / 64` for this fixed ARGB8888 linear case and use it for `surf0_ttu_cntl0/1`; keep `pitch_pixels` for pitch/PTE-row calculations.

- Note: Medium - `src/drivers/amd/atombios/dcn10_calc.c:9`, `58-67`, and `114-116` hard-code `DCN10_DEEP_SLEEP_DCFCLK_KHZ` as 8000. Linux DML does not use a fixed DCN1 constant here: `t_calc_us = 24 / deepsleep_dcfclk_mhz` and min TTU vblank comes from the DLG sys params (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/dml/dml1_display_rq_dlg_calc.c:1162-1170`, `1918-1919`), while the DCN1 IP table only supplies delay constants such as `dppclk_delay_subtotal = 90` and `dispclk_delay_subtotal = 42` (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/dml/dcn10/dcn10_fpu.c:95-99`). A too-low invented DCFCLK makes vupdate/vready/prefetch overly conservative and can reject valid modes; a too-high one risks underflow. Minimal fix: add/pass an actual deep-sleep DCFCLK kHz value from the programmed clock path, or document and enforce 8 MHz as a proven lower bound for the firmware environment.
