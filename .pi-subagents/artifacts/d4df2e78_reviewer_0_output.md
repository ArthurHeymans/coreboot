### Ranked flash-blocking findings

1. **Blocker: OTG blank/timing double-buffer initialization is inverted/missing.**  
   Current code writes `OTG_DOUBLE_BUFFER_CONTROL` with mask `0x00010100` and value `0x00000100` in `src/drivers/amd/atombios/dcn10.c:309-311`. Per Linux DCN1 register masks, bit 8 is `OTG_UPDATE_INSTANTLY` and bit 16 is `OTG_BLANK_DATA_DOUBLE_BUFFER_EN` (`/tmp/linux-6.12/drivers/gpu/drm/amd/include/asic_reg/dcn/dcn_1_0_sh_mask.h:21888-21900`), so this sets instant updates and clears blank-data double buffering. Linux initializes the TG the opposite way: `optc1_tg_init()` enables blank-data double buffering, enables timing double buffering, and clears OPTC underflow (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/optc/dcn10/dcn10_optc.c:1435-1439`), and `dcn10_init_pipes()` calls `tg_init()` during pipe init (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/hwss/dcn10/dcn10_hwseq.c:1477-1482`). With current sequencing, the final unblank at `src/drivers/amd/atombios/dcn10.c:562-563` is not vupdate-synchronized like Linux and can expose partially committed OTG/plane state.

2. **Blocker: underflow handling/validation is not equivalent to Linux and can both false-fail and miss real stream failure.**  
   Current code tries to clear HUBP underflow by clearing status bits in the DCHUBP_CNTL value before commit (`src/drivers/amd/atombios/dcn10.c:555-558`) and later tests the same status bits (`src/drivers/amd/atombios/dcn10.c:574-584`). Linux defines status at bits 28-30 and a separate clear bit at bit 31 (`/tmp/linux-6.12/drivers/gpu/drm/amd/include/asic_reg/dcn/dcn_1_0_sh_mask.h:9242-9252`), and clears by writing `HUBP_UNDERFLOW_CLEAR=1` (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/hubp/dcn10/dcn10_hubp.c:103-107`). Linux final underflow logic also checks/clears OPTC underflow before HUBP (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/hwss/dcn10/dcn10_hwseq.c:596-608`), while coreboot has no `OPTC_INPUT_GLOBAL_CONTROL` check/clear even though Linux unblank explicitly clears OPTC underflow (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/optc/dcn10/dcn10_optc.c:426-441`, `1428-1433`). This makes final validation unreliable for flashing: stale HUBP underflow can reject a good commit, and OPTC underflow can pass unnoticed.

3. **Blocker: FMT clamp programming contradicts Linux’s OPP path.**  
   Coreboot explicitly clears `FMT_CLAMP_CNTL` for the fixed RGB path (`src/drivers/amd/atombios/dcn10.c:531-534`). Linux still programs the OPP FMT path for active streams from core DC (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/core/dc.c:3309-3314`), and DCN1 `opp1_program_fmt()` calls clamping/pixel-encoding programming (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/opp/dcn10/dcn10_opp.c:300-318`). For full-range RGB, Linux enables clamp with color format 0 (`/tmp/linux-6.12/drivers/gpu/drm/amd/display/dc/opp/dcn10/dcn10_opp.c:209-221`); clearing the whole register leaves `FMT_CLAMP_DATA_EN=0` instead of Linux’s full-range clamp state.
